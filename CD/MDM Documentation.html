🏛️ Master Data Management — Foundations

What is MDM?

MDM is the practice of creating and maintaining a single, accurate, and trusted view of critical enterprise data — such as customers, products, suppliers, and locations — across all business systems.

The Core Problem MDM Solves

  • Same customer exists in SAP, Salesforce, and Oracle with different names/addresses
  • Same product has 5 different descriptions across eCommerce, ERP, and WMS
  • Business analytics are inaccurate due to duplicate and inconsistent records
  • Regulatory compliance (GDPR, CCPA) is difficult without a single customer view

Four MDM Implementation Styles

  • Registry: MDM holds only IDs + cross-references. No data stored. Low disruption.
  • Consolidation: MDM reads from source systems, creates a read-only golden record for analytics.
  • Coexistence: Golden record created AND published back to source systems (bidirectional).
  • Centralized (Hub): MDM is the system of record. All systems write to and read from MDM.

Core MDM Data Domains

  • Customer MDM — B2C and B2B customer master, household and account hierarchies
  • Product MDM — Product catalog, specifications, classifications
  • Supplier MDM — Vendor master, compliance, onboarding
  • Employee MDM — HR master aligned with HCM systems
  • Location MDM — Stores, sites, geographic hierarchies
  • Financial MDM — Chart of accounts, cost centers, legal entities
💡 Informatica MDM Hub specializes in Customer and Supplier MDM, while Informatica Product 360 focuses on Product MDM (PIM). Know which product covers which domain.

What is Informatica MDM Hub?

Deployment Options

  • Cloud (IICS) — Informatica Intelligent Cloud Services hosted MDM
  • Hybrid — On-prem Hub with cloud integration services

🗂️ MDM Hub Core Components

Hub Server

The central application server (deployed on JBoss/WebLogic/WebSphere). Hosts the core MDM business logic, ORS (Operational Reference Store), and all configuration metadata.

Process Server

Handles batch and real-time matching, merging, cleansing, and tokenization. A separate server for performance isolation. Runs match jobs and cleanse services.

Hub Console

Admin and configuration UI for MDM Hub. Used by developers and architects to configure base objects, match rules, mappings, workflows, and security. Java Swing desktop application.

Informatica Data Director (IDD)

Web-based data stewardship UI used by business users to review, edit, merge/unmerge records, manage hierarchies, and process workflow tasks.

Operational Reference Store (ORS)

The relational database schema (Oracle or SQL Server) that stores all MDM data — source records, cross-references (XREF), best version of record (BVR), and match tokens.

📌 The ORS is the heart of Informatica MDM. All source data lands in staging tables, gets cleansed and matched, and a golden record (Best Version of Record) is maintained alongside XREFs back to source systems.

📊 Base Objects & Columns

What is a Base Object?

A Base Object (BO) is the core entity in Informatica MDM. It maps to a table in the ORS database and represents the master record (e.g., PARTY, ACCOUNT, PRODUCT). The golden record lives here.

Key Base Object Tables

-- For a base object "PARTY", MDM creates: C_PARTY -- Base object table (golden/consolidated records) C_PARTY_XREF -- Cross-reference: source records linked to BO C_PARTY_PND -- Pending records (awaiting match/merge) C_PARTY_STRP -- Stripped/tokenized data for matching C_PARTY_HIST -- History of changes (audit trail) C_PARTY_MTCH -- Match result set from match jobs

Column Types

  • Standard Column — Regular data attribute (name, email, phone)
  • System Column — MDM-managed (ROWID_OBJECT, CREATE_DATE, UPDATED_BY)
  • Trust Column — Stores trust score per source per field
  • Period Column — Temporal data for effective dating

Trust & Survivorship

Each column can have a trust configuration. When multiple source systems provide the same field, the source with the highest trust score wins for the consolidated (golden) record.

🗃️ Staging Tables & Landing

Data Landing Flow

Source System → Landing Table → Staging Table → Cleanse → Match → Merge → Base Object (Golden)

Landing Tables (LT)

  • Raw staging area — exact copy of source data
  • Named: L_<BO_NAME>_<SOURCE> (e.g., L_PARTY_SAP)
  • Loaded via ETL (PowerCenter, IICS, or custom SQL)
  • Contains SOURCE_KEY (primary key from source system)

Staging Tables (ST)

  • Transformed and cleansed version of landing data
  • Named: S_<BO_NAME>_<SOURCE> (e.g., S_PARTY_SAP)
  • Cleanse functions applied here (address standardization, name parsing)
  • Source trust values assigned per column

Source System Configuration

Each source system is registered in MDM Hub Console with a unique system name. Trust levels, delta detection method (date-based or full), and merge protocol are configured per source.

🔗 Relationships & Hierarchies

Relationship Types

  • One-to-Many (1:N) — A PARTY has many ADDRESSes
  • Many-to-Many (M:N) — PARTY ↔ ORGANIZATION (via relationship table)
  • Self-Referencing — Household hierarchy (PARTY is parent of PARTY)
  • Foreign Key Relationship — ACCOUNT references PARTY

Hierarchy Management

Informatica MDM includes a Hierarchy Manager (HM) module for managing complex organizational hierarchies — parent companies, subsidiaries, household members, org charts.

  • Hierarchy Profiles — Define hierarchy types (Account, Household, Org)
  • Relationship Base Objects — Store hierarchy edges with effective dates
  • Hierarchy nodes can have relationship attributes (e.g., ownership percentage)
  • Time-aware hierarchies with effective start/end dates

Common Hierarchy Use Cases

  • Global Account Hierarchy: Global Corp → Regional HQ → Country → Site
  • Customer Household: Primary → Spouse, Children, Dependents
  • Supplier Network: Parent Supplier → Sub-vendors → Sub-contractors

🏷️ Lookup Tables & Code Lists

Lookup (Reference) Tables

Lookup tables store controlled vocabulary and reference data used across MDM. Examples: Country codes, State codes, Industry classifications, Customer types.

-- Example lookup table C_REPOS_CO_TYPE -- Company type lookup ROWID_OBJECT, CODE, NAME, DESCRIPTION 'CORP', 'Corporation', 'Standard Corporation' 'LLC', 'Limited Liability Company', 'LLC entity' 'PART', 'Partnership', 'Business Partnership'

Code Lists in IDD

  • Lookup tables surface as dropdowns in the IDD stewardship UI
  • Can be loaded from source systems or maintained manually in IDD
  • Support hierarchical lookups (Category → Subcategory)
  • Multi-language support for code descriptions

🎯 Match Rules & Rulesets

Match Rule Concepts

A match rule compares field values between records to determine if they represent the same real-world entity. Rules can be exact or fuzzy.

Match Rule Components

  • Match Column: The field being compared (e.g., FIRST_NAME, EMAIL)
  • Match Strategy: Algorithm used (Exact, SSA fuzzy, Soundex, Edit distance)
  • Weight: How much this column contributes to the overall score (0–100)
  • Null Handling: How to score when one or both values are null

Match Rule Sets

A Rule Set groups multiple match rules. A record pair must pass a Rule Set to be considered a match. You typically create multiple rule sets for different match scenarios:

Rule Set 1 — Exact ID Match (highest confidence): EMAIL = Exact → weight 100 → threshold 100 → Pass = Definite Match (auto-merge) Rule Set 2 — Fuzzy Name + Address: FIRST_NAME = SSA Fuzzy → weight 30 LAST_NAME = SSA Fuzzy → weight 40 ZIP_CODE = Exact → weight 30 → Score ≥ 85 = Probable Match (manual review) → Score 70–84 = Possible Match (steward review)

Match Thresholds

  • Auto-Merge Threshold — Score above this → records auto-merged (no human review)
  • Match Threshold — Score above this but below auto-merge → flagged for review
  • Below Match Threshold — Records are NOT duplicates

🔤 Tokenization & Standardization

What is Tokenization?

Before matching, MDM generates match tokens — normalized representations of field values that enable fast comparison at scale. Without tokens, matching millions of records would require N² comparisons.

Tokenization Strategies

  • SSA-Name3 — Informatica's proprietary phonetic/linguistic name matching engine
  • Soundex — Classic phonetic algorithm (Smith = Smyth)
  • Double Metaphone — Improved phonetic for multilingual names
  • Edit Distance (Levenshtein) — Handles typos and transpositions
  • Exact Token — For numeric IDs, emails (no normalization needed)

Cleanse Functions

Informatica MDM integrates with Informatica Data Quality (IDQ) or Address Doctor for cleansing before staging:

  • Address standardization and USPS/postal validation
  • Name parsing (First, Middle, Last, Suffix, Title)
  • Phone number normalization (E.164 format)
  • Email syntax validation
  • Gender and title inference from name

🏆 Survivorship & Trust

What is Survivorship?

When multiple source records are merged into one golden record, survivorship rules determine WHICH source value wins for each column.

Survivorship Strategies

  • Trust-based: Column value with highest trust score wins
  • Source Priority: ERP data > CRM data > Portal data (ranked)
  • Most Recent: Most recently updated value wins
  • Most Reliable (Non-null): First non-null value in priority order
  • Oldest: Original creation data preserved
  • Custom Function: Business logic (e.g., longest string, max value)

Trust Configuration

Source System Trust Config for FIRST_NAME column: SAP_ERP → Max Trust: 90, Decay: 30 days SALESFORCE → Max Trust: 75, Decay: 7 days WEB_PORTAL → Max Trust: 50, Decay: 1 day → Trust decays over time if not refreshed → SAP_ERP wins for FIRST_NAME unless Salesforce updated recently (recency boost)

Best Version of Record (BVR)

The BVR is Informatica's term for the golden record — the consolidated view combining the best value from each source for each column, per the survivorship rules.

🔀 Merge & Unmerge

Auto-Merge

When match score exceeds the auto-merge threshold, records are automatically merged without human intervention. The merged records share a single ROWID_OBJECT in C_<BO>.

Manual Merge (in IDD)

  • Data steward reviews potential match pairs in IDD
  • Can see side-by-side comparison of field values
  • Can select which source value to use for each field
  • Confirm or reject the merge

XREF Table (Cross-Reference)

After merge, the C_<BO>_XREF table maps source records back to the golden record:

C_PARTY_XREF: ROWID_OBJECT = 'GLD001' ← Golden record SYSTEM_NAME = 'SAP' SOURCE_KEY = 'SAP_CUST_4421' ← Source ID ROWID_OBJECT = 'GLD001' ← Same golden record SYSTEM_NAME = 'SFDC' SOURCE_KEY = '0034000000ABC' ← SFDC ID

Unmerge

Data stewards can unmerge incorrectly merged records. Unmerge restores source records to independent golden records and updates XREFs accordingly. Full audit trail maintained.

🖥️ Informatica Data Director (IDD)

What is IDD?

IDD is the web-based UI for business data stewards. It provides a business-friendly interface to search, view, edit, merge, unmerge, and manage master records — without needing to know SQL or the Hub Console.

Subject Areas

Subject Areas are IDD views configured to show related data together. Example: "Customer 360" Subject Area shows Party + Addresses + Phone + Email + Relationships on one screen.

  • Primary Object — The main base object (e.g., PARTY)
  • Children — Related objects displayed as sections (ADDRESS, PHONE)
  • Related Objects — Cross-referenced entities (ACCOUNT, ORDER)
  • Hierarchy View — Org/household hierarchy visualization

IDD Features

  • Search with faceted filters and saved searches
  • Side-by-side merge comparison screen
  • Timeline view of data changes
  • Task inbox for workflow assignments
  • Hierarchy drag-and-drop editor
  • Bulk edit and bulk merge/unmerge operations

🔄 Workflow & Task Management

ActiveVOS (Workflow Engine)

Informatica MDM uses ActiveVOS as its embedded BPM (Business Process Management) engine. Workflows are defined as BPEL processes and deployed to the ActiveVOS server.

Common MDM Workflows

  • Create Party Workflow — New record review before becoming master
  • Update Review — Significant field changes require approval
  • Match Review — Probable match pairs sent to steward for review
  • Merge Approval — Manual merge requires business sign-off
  • Data Exception — DQ rule violations routed to responsible steward

Workflow Task Types

  • Review and Approve — Steward reviews new/changed record
  • Merge and Unmerge — Confirm or reject match pair
  • Reject — Record returned to source with comments
  • Escalate — Route to supervisor/senior steward
  • Auto-Approve — Based on source trust or data completeness
📌 Workflows are configured in Hub Console under Process Manager, then deployed to ActiveVOS. IDD automatically shows workflow tasks in the steward's task inbox.

👥 Security & Roles

MDM Security Model

  • Users — Individual accounts managed in MDM or LDAP-synced
  • Roles — Collections of privileges assigned to user groups
  • Resource Roles — Permissions on specific base objects and operations
  • Column Security — Restrict read/write access per column per role

Key Built-in Roles

  • MDM Administrator — Full system access, Hub Console config
  • Data Steward — IDD access for record management and workflow
  • Data Manager — Can merge, unmerge, and override survivorship
  • Business User — Read-only view of golden records
  • Integration User — API access for source system integration

Sensitive Data Masking

Configure column-level security to mask SSN, credit card, or PII fields for roles that don't need full access. Masked columns show as ****** in IDD and API responses.

📋 Audit Trail & History

History Tables

For every base object, MDM maintains a C_<BO>_HIST table that records every change — who changed what, when, and from which source system.

  • CREATE_DATE, UPDATED_DATE, UPDATED_BY on every record
  • Full field-level change history with before/after values
  • Source system attribution for every field value
  • Merge/unmerge event audit in separate audit tables

Audit Configuration

  • Enable/disable history at the base object level
  • Configure history retention period (purge old history)
  • Selective column auditing — only audit sensitive or key fields
🔒 Audit trails are critical for GDPR/CCPA compliance — MDM must be able to show the full lineage of any data point and support "right to erasure" requests.

🔌 MDM REST & SOAP APIs (SIF)

Services Integration Framework (SIF)

SIF is Informatica MDM's API layer. It exposes operations for searching, creating, updating, merging, and managing records programmatically.

SIF REST API (MDM 10.x+)

-- Base URL https://<host>:<port>/cmx/cs/<ors-db>/ -- Key Endpoints GET /PARTY/<rowid> → Get record by ID GET /PARTY?searchToken=john → Search records POST /PARTY → Create new record PUT /PARTY/<rowid> → Update record POST /PARTY/<rowid>/XREF → Add cross-ref POST /merge?rowidObject1=<id1>&rowidObject2=<id2> → Merge POST /unmerge/<xref-rowid> → Unmerge -- Authentication: Basic Auth or SAML

SIF SOAP API

  • Legacy API, still widely used in older integrations
  • WSDL at: /cmx/services/SiperianClient?wsdl
  • Key operations: searchBOs, putBOs, getBOs, matchMerge

⚙️ PowerCenter & IICS Integration

Informatica PowerCenter (Traditional ETL)

  • Use PowerCenter mappings to extract from source systems
  • Transform and map to MDM Landing table format
  • Load into L_<BO>_<SOURCE> landing tables via JDBC
  • Trigger Hub batch jobs after load (Stage, Match, Merge, Promote)

IICS (Informatica Intelligent Cloud Services)

Cloud-native integration platform. Use IICS Mappings and Application Integration tasks to load MDM landing tables and call SIF APIs in real time.

MDM Hub Batch Jobs Sequence

1. Stage Job → Land → Stage (apply cleanse + trust) 2. Match Job → Generate match tokens + run match rules 3. Merge Job → Auto-merge above threshold 4. Promote Job → Promote consolidated data to base object 5. Notify Job → Fire outbound events/messages

🔁 Real-time & Batch Processing

Batch Processing

  • Scheduled via Hub Console Batch Viewer or external scheduler (Control-M, Autosys)
  • Best for: large volume initial loads, nightly delta loads
  • Full job sequences: Stage → Match → Merge → Promote → Publish
  • Can be triggered via JMX, SIF API, or external scheduler

Real-time Processing (SIF)

  • Source systems call SIF REST/SOAP API for single-record create/update
  • MDM Hub processes synchronously and returns golden record ID
  • Match is performed in real time for the new/updated record
  • Used for: web form submissions, CRM real-time sync, point-of-sale

Hybrid Pattern

Most enterprise MDM implementations use both: nightly batch for bulk loads from ERP/legacy systems AND real-time SIF API for customer-facing applications that need immediate golden record lookup.

📡 Events & Outbound Sync

MDM Event Framework

MDM Hub can publish events when data changes — record created, merged, promoted, or updated. Downstream systems subscribe to these events for real-time sync.

  • JMS Message Queue (ActiveMQ, WebLogic JMS) for async events
  • Event types: CREATED, UPDATED, MERGED, UNMERGED, PROMOTED
  • Event payload contains ROWID_OBJECT and changed field names
  • Consumer reads event and calls SIF API to fetch full golden record

Outbound Integration Patterns

  • Push to SAP ERP via IDoc or BAPI (PowerCenter or IICS)
  • Push to Salesforce via REST API or Salesforce connector
  • Publish to data lake / DWH via JDBC or file export
  • Real-time notification via Kafka topics (modern architecture)

🤖 CLAIRE AI Engine

What is CLAIRE?

CLAIRE (Cloud-scale, Learning, AI-driven, Recommendations, Execution) is Informatica's AI/ML engine embedded across the IDMC platform, including MDM.

CLAIRE Capabilities in MDM

  • ML-based match scoring — learns from steward accept/reject decisions
  • Automated match threshold recommendations
  • Data quality anomaly detection on incoming records
  • Survivorship suggestions based on historical patterns
  • Auto-classification of parties (B2B vs B2C, company type)
  • Duplicate detection confidence scoring with explainability

Training CLAIRE

The more stewards review and act on match pairs in IDD, the more CLAIRE learns. Steward decisions (accept/reject) feed back into the ML model, continuously improving match quality over time.

📐 Multi-Domain MDM Architecture

Single ORS vs Multiple ORS

  • Single ORS — All domains in one database schema (simpler, risk of contention)
  • Multiple ORS — Separate ORS per domain (Customer ORS, Supplier ORS) for isolation
  • Best practice: Separate ORS per major domain for enterprise implementations

Cross-Domain Relationships

Real power comes from linking domains. A Customer 360 links PARTY (Customer ORS) + PRODUCT (Product ORS) + SUPPLIER (Supplier ORS) to give a complete picture.

  • Customer purchased which Products (Customer ↔ Product)
  • Product manufactured by which Supplier (Product ↔ Supplier)
  • Customer is also a Supplier (B2B scenario)

Business 360 / Customer 360 SaaS

Informatica's Customer 360 SaaS is a pre-built multi-domain MDM solution (built on IDMC MDM) that provides ready-to-use data models, match rules, and IDD configurations for Customer, Account, and Household management.

⚡ Performance Tuning

Match Job Optimization

  • Ensure match index tables (C_<BO>_STRP) are properly indexed in DB
  • Tune match batch size — larger batches for high memory servers
  • Use parallel match processing across multiple Process Server threads
  • Implement match key segmentation to reduce match search space
  • Archive resolved match pairs to prevent match table bloat

Database Tuning

  • Partition large base object tables by ROWID range or date
  • Create composite indexes on frequently searched columns
  • Regular XREF table cleanup (remove orphaned XREFs)
  • Use Oracle RAC or SQL Server AlwaysOn for HA + read scaling

Load Balancing

  • Deploy multiple Process Servers for parallel cleanse/match jobs
  • Separate Hub Server (API) from Process Server (batch jobs)
  • Use connection pooling for SIF API high-concurrency scenarios

MDM Core Concepts — The Complete Picture

The MDM Golden Record

The golden record (also called Best Version of Record or BVR in Informatica) is the single, trusted, consolidated master record created by merging data from multiple source systems. It is NOT a copy of any one source — it picks the best value for each field.

XREF — The Magic Link

When two source records are merged into one golden record, the C_<BO>_XREF table maps source IDs back to the ROWID_OBJECT of the golden record. This is how MDM maintains referential integrity and allows source systems to be updated when master data changes.

Trust Score Decay

Informatica MDM implements temporal trust — a trust score decays over time if a source system doesn't refresh the value. A CRM that updates a phone number today has high trust; if it doesn't update for 6 months, its trust decays, and a fresher source can win survivorship.

Hub Console vs IDD — Know the Difference

Hub Console (Developer/Admin Tool)

  • Java Swing desktop application (not web-based)
  • Used by MDM developers and architects
  • Configure base objects, columns, relationships
  • Define match rules, cleanse functions, trust settings
  • Manage batch jobs, security, ORS configuration
  • Not used by business users in production

Informatica Data Director — IDD (Business User Tool)

  • Web-based application (browser-accessible)
  • Used by data stewards and business users
  • Search, view, edit, and manage golden records
  • Review and act on workflow tasks (merge/approve/reject)
  • Manage hierarchies (drag-and-drop)
  • No configuration access — business data only

Architecture

MDM Hub Technical Architecture

Architecture Layers

  • Database Layer — Oracle or SQL Server ORS schema (C_, S_, L_ tables)
  • Application Layer — Java EE Hub Server (JBoss/WebLogic) with SIF APIs
  • Process Layer — Process Server for cleanse, match, tokenization jobs
  • Workflow Layer — ActiveVOS BPM engine for approval workflows
  • Presentation Layer — IDD (Web UI) and Hub Console (Swing admin)
  • Integration Layer — SIF REST/SOAP, JMS events, PowerCenter/IICS

Data Flow Architecture

Source System → ETL (PowerCenter/IICS) → Landing Tables → Staging Tables → [Cleanse] → [Match Tokens] → [Match Job] → [Merge Job] → Base Object (Golden) → [Events/API] → Downstream Systems

High Availability Setup

  • Hub Server cluster (multiple nodes behind load balancer)
  • Process Server cluster (parallel match jobs)
  • Oracle RAC or SQL Server AlwaysOn for DB HA
  • Shared NFS for configuration artifacts
  • ActiveVOS in cluster mode for workflow HA

Technical

ORS Schema — What's in the Database

System Tables (C_REPOS_*)

MDM stores all configuration metadata in C_REPOS_* tables within the ORS. This includes base object definitions, column metadata, match rule configurations, and source system registrations.

⚠️ Never directly modify C_REPOS_* tables. All configuration must be done through Hub Console. Direct DB changes can break the MDM configuration.

Database Requirements

  • Oracle 19c/21c — Recommended, full feature support
  • Microsoft SQL Server 2019+ — Supported, some limitations
  • Minimum 200GB+ for ORS on production (depends on record volume)
  • Separate tablespaces for data, indexes, LOB, and temp

Match & Merge — Complete Reference

Why Match & Merge is the Hardest Part

Getting match rules right is the most difficult and most important skill in MDM. Match too aggressively → false merges (two different people merged into one). Match too conservatively → duplicates remain. The right balance requires domain knowledge and testing with real data.

Match Strategy Comparison

StrategyBest ForFalse Positive Risk
ExactIDs, emails, phone numbersVery Low
SSA-Name3 FuzzyPerson names (phonetic)Medium
SoundexSimple name phoneticsMedium-High
Edit DistanceTypos, abbreviationsHigh
Address FuzzyAddress line matchingMedium
Custom FunctionBusiness-specific logicConfigurable

Match Rule Design Best Practices

  • Always start with the highest-confidence rule set (exact ID match)
  • Layer fuzzy rules with multiple corroborating fields (name + DOB + ZIP)
  • Never auto-merge on name alone — always combine with address or ID
  • Test rules with a sample of known duplicates before production
  • Set a "possible match" band for steward review (not just auto-merge vs no-match)
  • Review and tune thresholds every 3–6 months based on match quality metrics

Null Handling in Match Rules

  • Null-Null → typically score as 0 (unknown match)
  • Value-Null → typically penalize score by 50% (one side unknown)
  • Configure per-column based on data completeness expectations

Survivorship

Survivorship Design Patterns

Designing Survivorship Rules

Survivorship should be a business decision, not a technical one. Sit with business stakeholders and ask: "If CRM and ERP have different values for this field, which do you trust more?"

Common Survivorship Patterns

  • Legal Name: ERP (official records) > CRM (sales-entered)
  • Email Address: CRM > ERP (CRM is customer-facing, more accurate)
  • Tax ID / Reg Number: ERP only (single authoritative source)
  • Address: Address Doctor standardized value > raw source
  • Phone: Most recently updated source wins (recency-based)
  • Customer Since Date: Oldest value wins (preserve original relationship)

Trust Decay Formula

Trust at time T = Max Trust × e^(-decay_rate × days_since_update). Configure decay rates per source per column to reflect how quickly the source's data becomes stale.
