🏗️ What is MDM?

Definition

Master Data Management (MDM) is a technology-enabled discipline in which business and IT work together to ensure the uniformity, accuracy, stewardship, semantic consistency, and accountability of the enterprise's shared master data assets.

Core MDM Types

  • Product MDM – Manage product hierarchies, attributes, relationships
  • Customer MDM – Single view of customer across systems
  • Supplier MDM – Vendor onboarding, compliance, and data quality
  • Location MDM – Store/site hierarchies and geographic data
  • Employee MDM – HR master data aligned with ERP/HCM

MDM Implementation Styles

  • Registry Style – Central index, data stays in source systems
  • Consolidation Style – Read-only golden record created from sources
  • Coexistence Style – Golden record published back to source systems
  • Centralized Style – MDM is the system of record (STIBO STEP)

Business Benefits

  • Single source of truth for critical data
  • Reduced data inconsistency across systems (ERP, CRM, PIM)
  • Faster time-to-market for products
  • Better regulatory compliance and data governance
  • Improved analytics and reporting accuracy

🧱 Object Types & Hierarchies

Object Types

Object types define the schema for your data entities. Each object type specifies which attributes, references, and validations apply.

  • Product Object Types: Base Product, SKU, Bundle, Kit
  • Classification Object Types: Category, Subcategory nodes
  • Entity Object Types: Customer, Supplier, Location, Asset

Hierarchy Types in STEP

  • Product Hierarchy – Physical parent-child product relationships
  • Classification Hierarchy – Logical taxonomy for grouping (eCl@ss, UNSPSC, GS1)
  • Specification Hierarchy – Defines attribute inheritance paths
  • Global Specification – Attributes visible across all contexts

Inheritance

STEP supports attribute inheritance from parent nodes to children. Configure inheritance rules to avoid redundant data entry.

Parent: Electronics
├── Voltage: 220V (inherited by all children)
└── Child: Laptops
└── Child: Gaming Laptops
└── GPU Type: RTX 4090 (specific attribute)

📋 Attributes & Attribute Groups

Attribute Data Types

  • Text – Single-line or multi-line free text
  • Number – Integer or decimal with unit of measure
  • Boolean – Yes/No flag
  • Date/DateTime – Date picker with timezone support
  • LOV (List of Values) – Single or multi-select dropdown
  • External Source LOV – Dynamic LOV from external system
  • Asset Cross Reference – Link to digital asset

Attribute Validations

  • Mandatory / Optional / Conditionally Required
  • Regular Expression pattern matching
  • Min/Max value constraints
  • Length constraints for text fields
  • Custom Business Rule validations (JavaScript/Groovy)

Context-Specific Attributes

Attributes can be global (same across all contexts) or context-specific (different values for EN-US vs DE-DE vs FR-FR). This is critical for multilingual PIM implementations.

🔗 References & Relationships

Reference Types

  • Product-to-Product: Cross-sell, upsell, spare parts, accessories
  • Product-to-Asset: Main image, additional images, documents
  • Product-to-Classification: Category assignment
  • Product-to-Supplier: Manufactured by, supplied by
  • Customer-to-Customer: Parent company, subsidiary

Reference Cardinality

  • One-to-One (1:1) – A product has one primary supplier
  • One-to-Many (1:N) – A product has many images
  • Many-to-Many (M:N) – Products can belong to multiple categories

Reference Attributes

References themselves can carry attributes! e.g., a "Supplied By" reference can have attributes like "Lead Time", "Contract Start Date", "Preferred Supplier: Yes/No".

🌍 Contexts & Workspaces

Contexts

A Context = Language + Channel. STEP allows you to maintain different attribute values per context, enabling true multichannel, multilingual data management.

Context Examples:
EN-US-WEB → English, US, Web Channel
DE-DE-PRINT → German, Germany, Print Catalog
FR-FR-APP → French, France, Mobile App
GLOBAL → Language-independent master data

Workspaces

  • Main Workspace – Working / draft version of data
  • Approved Workspace – Published, approved data
  • Custom Workspaces – Staging, Review, QA environments

Workspace Approval Flow

Data changes in Main workspace go through workflow approval before promotion to Approved workspace. The Approved workspace feeds downstream systems via exports.

⚖️ Business Rules Engine

Types of Business Rules

  • Validation Rules – Check if data meets criteria before save
  • Calculated Attributes – Auto-compute values from other fields
  • Transformation Rules – Normalize or transform incoming data
  • Event Rules – Trigger actions on create, update, delete
  • Completeness Rules – Measure how complete a record is

Implementation Languages

  • JavaScript (STEP Business Function) – Most common
  • Groovy – For complex server-side logic
  • STEP Expression Language – For simple conditional rules

Example Business Rule

// Validation: Product weight must be positive
function validate(node) {
var weight = node.getValue('ProductWeight');
if (weight !== null && weight <= 0) {
throw new Error('Weight must be greater than 0');
}
}

👥 Users, Roles & Privileges

STEP Security Model

  • Users – Individual accounts with role assignments
  • Groups – User groups mapped to roles
  • Roles – Collections of privileges (read, write, admin)
  • Privileges – Granular permissions on objects, attributes, workflows

Privilege Types

  • Object Type Privileges – Can create/read/update/delete object types
  • Attribute Privileges – Read-only vs editable per attribute group
  • Workflow Privileges – Can initiate or complete workflow tasks
  • Export/Import Privileges – Access to data integration features
  • Admin Privileges – System configuration access

Data Stewardship

STEP supports defining data stewards per domain, object type, or classification. Stewards own data quality and approve changes in their domain.

📊 Data Quality & Completeness

Completeness Scoring

STEP calculates a completeness score for each object based on mandatory and optional attribute population. Scores can be context-specific.

  • Define completeness groups per object type
  • Assign weights to attributes (e.g., Title = 20%, Description = 30%)
  • Set thresholds for "ready for export" (e.g., Score ≥ 80%)

Data Quality Rules

  • Uniqueness checks – No duplicate product codes
  • Format validations – EAN-13 barcode format, email format
  • Referential integrity – Referenced objects must exist
  • Cross-field validation – If Type=Hazardous, then SafetyCode must be set

🔄 Smart Data Pool & GDSN

What is GDSN?

Global Data Synchronization Network (GDSN) is a GS1 standard for sharing product data between suppliers and retailers in a standardized format. STIBO's Smart Data Pool is a GS1-certified GDSN data pool.

  • Suppliers publish product data to the data pool
  • Retailers subscribe and receive standardized product records
  • Supports GPC (Global Product Classification)
  • CIC (Confirmation of Item Change) and CIN notifications

STEP GDSN Integration

  • STEP can act as publisher (outbound GDSN) or subscriber (inbound GDSN)
  • Automatic mapping from STEP attributes to GDSN data segments
  • GS1 validation rules enforced on publish
  • Smart Data Pool connector in STEP marketplace

📐 Matching & Deduplication

Match Rules

  • Exact match on identifier (GTIN, EAN, SKU)
  • Fuzzy match on name + category + brand
  • Phonetic match for customer name matching
  • Address standardization and matching
  • Machine learning match scoring (probabilistic matching)

Survivorship Rules

When multiple records match, survivorship rules determine which source wins for each attribute:

  • Most recent wins – Use latest updated value
  • Source priority – ERP > CRM > Portal (ranked trust)
  • Most complete wins – Non-null value preferred
  • Manual review – Flag for human decision

Golden Record

The golden record is the merged, authoritative master record. In STEP, linked source records are maintained alongside the golden record for lineage tracking and audit.

🏗️ MDM Core Concepts Fundamentals

The MDM Problem

In enterprises, the same data lives in many systems: a product exists in SAP (ERP), Salesforce (CRM), Magento (eCommerce), and warehouse systems — often with different names, formats, and values. MDM solves this by creating ONE trusted version.

The Four MDM Styles

  • Registry: MDM holds only identifiers and links to source systems. Low disruption but less control.
  • Consolidation: MDM creates a read-only golden record from source data. Good for analytics.
  • Coexistence: Golden record created AND published back to source systems. Bidirectional sync.
  • Centralized (Hub): MDM IS the system of record. All changes go through MDM. STIBO STEP uses this.

Why STIBO STEP?

  • Multidomain: Product + Customer + Supplier + Location in ONE platform
  • Configurable: Business users can define data model without coding
  • Enterprise-grade: Used by Walmart, Carlsberg, LEGO, Kmart, and 1000+ global brands
  • GS1 Certified: GDSN data pool certification for retail/CPG industries

🔑 Essential STEP Concepts Must Know

The STEP Object Model

Everything in STEP is a "node" in a tree. Nodes have: an Object Type (what kind of thing it is), Attributes (data fields), and References (relationships to other nodes).

Think of it like a database: Object Types = Tables, Attributes = Columns, References = Foreign Keys — but with hierarchy and versioning built in.

Workspace Versioning

STEP uses workspaces like Git branches. You edit data in "Main" workspace, it gets reviewed and promoted to "Approved" workspace. Only Approved data gets exported to downstream systems.

Context = Language × Channel

A single product can have different descriptions per context. "EN-US-WEB" might have a marketing description. "DE-DE-PRINT" has a catalog description in German. Both live in the same product record.

🗂️ Database & Storage Architecture Technical

STEP Database Schema

STEP uses a proprietary tree-based data model stored in RDBMS. Key tables include Nodes (all objects), Attributes (attribute definitions), Values (attribute values), and References (object relationships).

⚠️ Never directly query or modify the STEP database. Always use STEP APIs, exports, or the Workbench. Direct DB manipulation can corrupt the STEP data model.

Digital Asset Management (DAM)

  • Assets (images, videos, PDFs) stored in external file storage
  • STEP stores metadata + file path reference
  • Rendition generation for thumbnails, web-optimized versions
  • Asset usage tracking — which products use which assets

🔌 STEP REST API Deep Dive API

Authentication

  • Basic Authentication: Base64(username:password) in Authorization header
  • OAuth 2.0: Recommended for production — client credentials or auth code flow
  • API Keys: Available in some STEP versions for service accounts

Query Language (OData)

GET /rest-api/objects
?$filter=ObjectType eq 'Product'
&$filter=Attribute/Color eq 'Blue'
&$select=ID,Name,Color,Weight
&$top=100
&$skip=200
&$orderby=Name asc

Pagination & Rate Limits

  • Default page size: 50 records. Max: 500 records per request
  • Use $top and $skip for pagination
  • Rate limits vary by deployment (typically 100 req/min)
  • Use bulk import endpoints for large data loads instead of per-record API calls

🛠 Data Modeling Best Practices Best Practices

Golden Rules of STEP Data Modeling

  • Design the classification hierarchy FIRST — it drives everything else
  • Use inheritance to avoid repeating attribute assignments
  • Keep attribute names business-friendly, not IT-friendly (use business terms)
  • Plan contexts upfront — adding contexts later is painful
  • Always use LOVs for controlled vocabularies — never free text for standardized fields
  • Separate "global" (language-independent) from "contextual" attributes carefully

Hierarchy Design Patterns

  • Product Hierarchy: Family → Category → Subcategory → Product → SKU
  • Geographic: Region → Country → City → Location
  • Customer: Enterprise → Division → Department → Contact
  • Supplier: Parent Company → Subsidiary → Site → Contact

Common Modeling Mistakes

  • ❌ Creating too many object types (keep it simple)
  • ❌ Mixing classification and specification hierarchies
  • ❌ Using free text for fields that should be LOVs
  • ❌ Not planning for future domains (adding later requires redesign)
  • ❌ Ignoring workspace strategy until go-live
